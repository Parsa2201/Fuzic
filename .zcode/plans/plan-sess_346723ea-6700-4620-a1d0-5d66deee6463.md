## Goal
Downloads tab shows the downloaded songs list **if the user is Premium**, otherwise shows a **purchase (upgrade) page**. Clicking the upgrade button flips the user to Premium (demo only — no real purchase/billing flow), matching the PDF spec (Persian): "clicking the purchase button runs a demo flow and sets Premium to `true`."

## Approach
Gate the Downloads screen on the observed premium status, and reuse the existing demo purchase plumbing in `PremiumRepository` (already implemented in `RemotePremiumRepository` — `observePremiumStatus()` + `purchasePlan()` flip `is_premium` to true in Supabase). No new repository or backend work.

## Why this approach
- **DRY**: reuses `PremiumRepository` (already does demo purchase) and shared premium UI components.
- **Decoupled**: Downloads keeps owning its tab logic; no navigation to the separate Premium screen needed (Downloads is a bottom tab).
- **No backend work**: matches "we don't need purchase flow yet."
- **Minimal blast radius**: `PremiumRepository` interface is unchanged, so `FakePremiumRepository` and `PremiumViewModelTest` stay intact.

## Notes / trade-offs
- Plan-selection is intentionally skipped (no real purchase flow). Yearly plan is hardcoded as the demo entitlement; this is acceptable per the spec.
- Shared component extraction (`PremiumHeroCard`) lightly touches `PremiumScreen.kt` to consume it — small, keeps DRY.
